# GH-25 — [TASK-007] Rate limiting sur les API Routes

## Contexte
Partie de l'EPIC 5 Infrastructure (GH-5). Protège les API Routes contre les abus, les dérapages de coût IA et l'épuisement des quotas FMP.

## Description
En tant que développeur, je veux un rate limiting sur les API Routes critiques, afin de protéger contre les abus et les dérapages de coût.

## Ce qui doit être fait
- [ ] Créer `src/lib/rate-limiter.ts` — logique de rate limiting en mémoire (ou Supabase)
- [ ] Appliquer sur `/api/quote` — max 30 req/min
- [ ] Appliquer sur `/api/agents/*` (analyse, chat) — max 20 req/jour

## Fichiers concernés
| Fichier | Action |
|---------|--------|
| `src/lib/rate-limiter.ts` | créer — module rate limiting |
| `src/app/api/quote/route.ts` | modifier — appliquer rate limiter |
| `src/app/api/analyse/chat/route.ts` | modifier — appliquer rate limiter |
| `src/app/api/analyse/classic/route.ts` | modifier — appliquer rate limiter |
| `src/app/api/analyse/ticker/route.ts` | modifier — appliquer rate limiter |

## Critères d'acceptation
- [ ] CA1 : `GET /api/quote` retourne HTTP 429 après 30 requêtes en moins d'une minute
- [ ] CA2 : `POST /api/analyse/*` retourne HTTP 429 après 20 requêtes dans la journée
- [ ] CA3 : La réponse 429 inclut un header `Retry-After` avec le délai avant retry
- [ ] CA4 : Les limites sont par utilisateur authentifié (user_id depuis session Supabase)

## Hors périmètre
- Le rate limiting sur les routes d'écriture (positions, transactions)
- Une interface d'administration des limites
- Un dashboard de monitoring des quotas

## Notes techniques
- Pour Vercel (serverless) : le rate limiting en mémoire ne persiste pas entre les invocations
- Solution recommandée : compteurs dans Supabase (table `rate_limit_counters`) ou Vercel KV
- Décision d'implémentation à confirmer avec l'architect si Vercel KV n'est pas configuré
- Utiliser `src/lib/supabase/server.ts` pour les compteurs Supabase
