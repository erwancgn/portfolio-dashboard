# GH-91 — [US-IA-02] Structured output Gemini (JSON mode natif)

## Contexte

Toutes les routes d'analyse extraient le JSON des réponses Gemini avec une regex (`extractLastJsonObject()`). C'est fragile : si le modèle enveloppe le JSON dans du markdown, change l'ordre, ou ajoute du texte après, l'extraction échoue silencieusement.

Gemini 2.5 supporte nativement `responseMimeType: "application/json"` + `responseSchema` qui garantit un JSON valide structuré sans regex.

## Description

En tant que développeur, je veux que les réponses JSON des analyses soient garanties structurées nativement par Gemini, sans extraction fragile par regex.

## Ce qui doit être fait

- [ ] Définir les schémas Zod/JSON Schema pour chaque type de réponse :
  - `BUFFETT_SCHEMA` (signal, score, moat, margin_of_safety, verdict)
  - `LYNCH_SCHEMA` (signal, score, peg, category, story, verdict)
  - `FAIR_VALUE_SCHEMA` (fair_value, current_price, signal, upside_percent, confidence, methodology)
  - `TICKER_SCHEMA` (signal, score)
- [ ] Activer `responseMimeType: "application/json"` + `responseSchema` dans `AIService.generate()` quand `expectJson: true`
- [ ] Supprimer `extractLastJsonObject()` des routes qui utilisent le mode structuré
- [ ] Conserver `extractLastJsonObject()` uniquement pour le chat (réponse markdown libre)
- [ ] Mettre à jour `ai-validation.ts` : les validators restent mais valident une structure déjà parsée (plus de try/catch JSON.parse)

## Critères d'acceptation

- [ ] CA1 : Les routes classic, fair-value, ticker retournent un JSON structuré sans regex
- [ ] CA2 : Les validators `ai-validation.ts` reçoivent un objet déjà parsé
- [ ] CA3 : La route chat reste inchangée (markdown libre)
- [ ] CA4 : Aucune régression sur les analyses (signal, score, verdict corrects)
- [ ] CA5 : En cas de schéma invalide, l'erreur est propagée proprement (pas de fallback silencieux)
