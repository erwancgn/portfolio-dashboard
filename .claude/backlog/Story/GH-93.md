# GH-93 — [US-IA-04] Quota tokens server-side (Supabase)

## Contexte

Le quota de tokens IA est actuellement suivi côté client : le navigateur envoie `tokensUsedToday` dans chaque requête et le serveur fait confiance à ce chiffre. Un utilisateur peut simplement envoyer `tokensUsedToday: 0` pour contourner la limite. Pour une appli personnelle ce n'est pas critique, mais si l'appli est partagée ou deployée, c'est un vecteur de coût non contrôlé.

## Description

En tant qu'administrateur, je veux que le quota de tokens soit suivi et enforced côté serveur, indépendamment de ce que le client envoie.

## Ce qui doit être fait

- [ ] Créer une table Supabase `user_token_usage` :
  ```sql
  user_id UUID REFERENCES auth.users(id),
  date DATE DEFAULT CURRENT_DATE,
  tokens_used INTEGER DEFAULT 0,
  PRIMARY KEY (user_id, date)
  ```
- [ ] Modifier `AIService.generate()` pour incrémenter `tokens_used` après chaque appel réussi
- [ ] Modifier `/api/analyse/chat/route.ts` : vérifier quota serveur avant d'appeler le LLM
- [ ] Supprimer la dépendance à `tokensUsedToday` depuis le body de la requête chat
- [ ] Retourner dans la réponse : `tokensUsedToday` (pour affichage UI)
- [ ] RLS : chaque utilisateur ne peut lire/écrire que sa propre ligne

## Critères d'acceptation

- [ ] CA1 : Quota dépassé → HTTP 429 même si le client envoie `tokensUsedToday: 0`
- [ ] CA2 : Compteur tokens incrémenté correctement après chaque appel chat
- [ ] CA3 : L'UI reçoit `tokensUsedToday` dans la réponse (pas de changement UX visible)
- [ ] CA4 : RLS empêche un user de lire les tokens d'un autre
