# GH-95 — [US-IA-06] Fast-track dev pour tickets pré-définis

## Contexte

Le workflow BMAD complet (analyst → pm → architect → sm → dev → review → QA) est conçu pour les nouvelles features à partir d'un besoin vague. Mais quand un ticket existe déjà dans le backlog avec des CAs et une estimation, relancer analyst+pm+sm revient à payer Sonnet pour re-lire ce qui est déjà écrit.

Coût actuel d'un appel inutile : ~$0.20-0.40 × 3 agents = ~$0.80 gaspillés par ticket pré-défini.

## Description

En tant que PO, je veux que l'équipe d'agents saute les phases d'analyse/découpage quand un ticket existe déjà, et passe directement au développement.

## Ce qui doit être fait

- [ ] Mettre à jour `agent-protocol.md` : ajouter règle "Fast-track" :
  > Si le ticket est dans `.claude/backlog/Story/` ou `.claude/backlog/Task/`, sauter analyst/pm/sm. Aller directement à : architect (si complexité haute) → dev-agent → code-reviewer → test-agent.
- [ ] Mettre à jour le skill `dev-workflow` pour détecter si un GH-XXX est fourni et si le fichier existe dans le backlog
- [ ] Ajouter dans `dev-workflow` : lecture du fichier ticket en premier (pas exploration générale)
- [ ] Documenter dans `LESSONS.md` : différence new-feature (BMAD complet) vs ticket-existant (fast-track)
- [ ] Clarifier quand architect est optionnel en fast-track : sauter si Story de taille S/M sans impact DB ni architecture

## Critères d'acceptation

- [ ] CA1 : Quand le PO dit "implémente GH-82", le workflow lit le ticket et lance dev-agent directement
- [ ] CA2 : analyst, pm, sm ne sont PAS invoqués pour un ticket backlog existant
- [ ] CA3 : architect est invoqué seulement si le ticket est de taille L ou implique DB/architecture
- [ ] CA4 : `dev-workflow` documente le chemin pris (fast-track ou BMAD) dans sa réponse initiale
