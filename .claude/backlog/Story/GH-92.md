# GH-92 — [US-IA-03] Cache données FMP (24h)

## Contexte

Les routes `/api/analyse/classic` et `/api/analyse/ticker` fetchent les données FMP (bilan, ratios, profil) à chaque appel, sans cache. Les données fondamentales (bilans, ratios) ne changent pas en intra-journalier — elles sont mises à jour trimestriellement. C'est +300-500ms de latence inutile par requête, et ça consomme le quota FMP (250 appels/jour).

## Description

En tant qu'utilisateur, je veux que l'analyse d'un ticker soit plus rapide, sans recharger les données fondamentales si elles ont déjà été récupérées aujourd'hui.

## Ce qui doit être fait

- [ ] Créer une table Supabase `fmp_data_cache` :
  ```sql
  ticker TEXT PRIMARY KEY,
  profile JSONB,
  financials JSONB,
  ratios JSONB,
  fetched_at TIMESTAMPTZ,
  expires_at TIMESTAMPTZ
  ```
- [ ] Créer `src/lib/fmp-cache.ts` avec `getCachedFmpData(ticker)` et `cacheFmpData(ticker, data)`
- [ ] TTL : 24h pour profil + ratios, 7j pour bilans (trimestriels)
- [ ] Intégrer dans les routes classic et ticker : vérifier cache avant appel FMP
- [ ] Les cours live (`/quote`) restent sans cache (intra-journalier)

## Critères d'acceptation

- [ ] CA1 : Second appel pour un même ticker le même jour ne fait pas de requête FMP
- [ ] CA2 : Cache expiré → fetch FMP + mise à jour cache
- [ ] CA3 : Les cours live ne sont jamais cachés
- [ ] CA4 : Quota FMP visible réduit sur appels répétés (vérifiable dans les logs)
- [ ] CA5 : Migration Supabase sans impact sur les données existantes
