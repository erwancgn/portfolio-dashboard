# GH-38-S1 — [US-COR-01] Soft delete d'une position

## Contexte
Partie de l'EPIC 12 (GH-38) Corbeille & suppression douce. Aujourd'hui, `DELETE /api/positions/[id]` effectue un hard delete irréversible (`.delete()` Supabase). Cette story remplace ce comportement par un soft delete : la position reste en base avec `deleted_at = now()` et n'apparaît plus dans le portfolio.

## Description
En tant que PO, je veux que la suppression d'une position la déplace en corbeille plutôt que de l'effacer définitivement, afin de pouvoir la restaurer en cas d'erreur.

## Ce qui doit être fait
- [ ] Modifier `DELETE /api/positions/[id]` : remplacer le `.delete()` par `.update({ deleted_at: new Date().toISOString() })`
- [ ] Vérifier que `GET /api/positions` ne retourne plus les positions soft-deleted (filtre `deleted_at IS NULL` via RLS ou requête explicite)
- [ ] Ajouter le filtre `.is('deleted_at', null)` dans `GET /api/positions` comme double protection si la RLS n'est pas suffisante
- [ ] S'assurer que les transactions liées à la position ne sont pas supprimées

## Fichiers concernés
| Fichier | Action |
|---------|--------|
| `src/app/api/positions/[id]/route.ts` | modifier — fonction `DELETE` : hard delete → soft delete |
| `src/app/api/positions/route.ts` | modifier — fonction `GET` : ajouter filtre `.is('deleted_at', null)` |

## Critères d'acceptation
- [ ] CA1 : `DELETE /api/positions/[id]` retourne 204 et positionne `deleted_at = now()` sur la ligne (pas de suppression physique)
- [ ] CA2 : La position soft-deleted n'apparaît plus dans la réponse de `GET /api/positions`
- [ ] CA3 : La position soft-deleted est toujours présente en base (vérifiable via `GET /api/positions/trash` — story suivante)
- [ ] CA4 : Les transactions liées à la position restent intactes en base après soft delete
- [ ] CA5 : Un `DELETE` sur un `id` inexistant ou appartenant à un autre utilisateur retourne toujours 404

## Hors périmètre
- La route `GET /api/positions/trash` (GH-38-S2)
- La route `POST /api/positions/[id]/restore` (GH-38-S2)
- La route `DELETE /api/positions/trash` pour purge définitive (GH-38-S2)
- Toute modification de l'interface utilisateur au-delà du comportement de suppression existant

## Dépendances

## Notes techniques
- Fichier à modifier : `src/app/api/positions/[id]/route.ts` — fonction `DELETE` ligne 90–131
- Pattern actuel : `.delete({ count: 'exact' }).eq('id', id).eq('user_id', user.id)`
- Pattern cible : `.update({ deleted_at: new Date().toISOString() }, { count: 'exact' }).eq('id', id).eq('user_id', user.id).is('deleted_at', null)`
- Le filtre `.is('deleted_at', null)` sur le UPDATE garantit qu'on ne "re-soft-delete" pas une position déjà en corbeille (idempotence)
- Utiliser `src/lib/supabase/server.ts` (déjà le cas dans ce fichier)
