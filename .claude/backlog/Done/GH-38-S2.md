# GH-38-S2 — [US-COR-02] Vue corbeille et restauration

## Contexte
Partie de l'EPIC 12 (GH-38) Corbeille & suppression douce. Une fois le soft delete en place (GH-38-S1), cette story expose la corbeille à l'utilisateur : consultation des positions supprimées, restauration individuelle, et purge définitive avec confirmation. Complète le cycle du soft delete côté utilisateur.

## Description
En tant que PO, je veux accéder à une corbeille listant mes positions supprimées, pouvoir les restaurer ou les effacer définitivement, afin d'avoir un filet de sécurité contre les suppressions accidentelles.

## Ce qui doit être fait
- [ ] Créer `GET /api/positions/trash` — liste les positions `deleted_at IS NOT NULL` de l'utilisateur
- [ ] Créer `POST /api/positions/[id]/restore` — remet `deleted_at = NULL` sur une position
- [ ] Créer `DELETE /api/positions/trash` — hard delete définitif de toutes les positions soft-deleted de l'utilisateur
- [ ] Créer le composant `src/components/positions/TrashDrawer.tsx` (ou section dans la page portfolio) affichant la liste des positions supprimées
- [ ] Afficher pour chaque position : ticker, nom, date de suppression formatée
- [ ] Bouton "Restaurer" par ligne
- [ ] Bouton "Vider la corbeille" avec modale de confirmation avant le hard delete
- [ ] État vide explicite si la corbeille ne contient aucune position
- [ ] Point d'entrée accessible depuis l'interface principale (bouton ou lien)

## Fichiers concernés
| Fichier | Action |
|---------|--------|
| `src/app/api/positions/trash/route.ts` | créer — GET (liste corbeille) + DELETE (purge définitive) |
| `src/app/api/positions/[id]/restore/route.ts` | créer — POST restauration |
| `src/components/positions/TrashDrawer.tsx` | créer — composant vue corbeille |
| `src/app/api/positions/route.ts` | vérifier — pas de modification attendue |

## Critères d'acceptation
- [ ] CA1 : `GET /api/positions/trash` retourne 200 avec la liste des positions `deleted_at IS NOT NULL` de l'utilisateur connecté
- [ ] CA2 : La vue corbeille est accessible depuis l'interface principale (bouton visible)
- [ ] CA3 : Chaque ligne de la corbeille affiche le ticker, le nom et la date de suppression (format lisible, ex: "31 mars 2026")
- [ ] CA4 : `POST /api/positions/[id]/restore` retourne 200, remet `deleted_at = NULL`, la position réapparaît dans le portfolio
- [ ] CA5 : `DELETE /api/positions/trash` retourne 204 et supprime définitivement toutes les positions soft-deleted de l'utilisateur
- [ ] CA6 : Le bouton "Vider la corbeille" déclenche une modale de confirmation avant l'appel `DELETE /api/positions/trash`
- [ ] CA7 : Si la corbeille est vide, la vue affiche un état vide explicite (texte ou illustration)
- [ ] CA8 : Un utilisateur ne peut pas voir ni restaurer les positions soft-deleted d'un autre utilisateur (double protection `user_id`)

## Hors périmètre
- Historique des suppressions (date + motif) au-delà de `deleted_at`
- Notifications ou alertes à la suppression
- Soft delete sur les transactions (seules les positions sont concernées)
- Pagination de la corbeille

## Dépendances
- Bloquée par : GH-38-S0 (colonne `deleted_at`), GH-38-S1 (soft delete opérationnel)
- Débloque : aucune (dernière story de l'epic)

## Notes techniques
- La route `src/app/api/positions/trash/route.ts` gère GET et DELETE sur la même URL `/api/positions/trash` — pattern identique à `src/app/api/positions/route.ts`
- La route restore crée un sous-dossier : `src/app/api/positions/[id]/restore/route.ts`
- Utiliser `src/lib/supabase/server.ts` dans toutes les routes API
- Composant `TrashDrawer.tsx` : `'use client'` requis (interactions boutons, modale)
- Format date suppression : utiliser `Intl.DateTimeFormat` avec locale `fr-FR`
- Le `DELETE /api/positions/trash` doit filtrer sur `user_id = user.id AND deleted_at IS NOT NULL` pour ne jamais supprimer des positions actives
